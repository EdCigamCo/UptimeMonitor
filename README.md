# UptimeMonitor

## Описание

Эта версия проекта добавляет фоновый воркер, который периодически проверяет доступность всех сохраненных сайтов и сохраняет результаты проверок в базу данных.

## Что добавлено

### Новые компоненты

1. **Расширенный Worker** (`infrastructure/worker/worker.go`):
    - Структура `Worker` с полями:
        - `db *sql.DB` - подключение к базе данных
        - `checkInterval time.Duration` - интервал между проверками
        - `ctx context.Context` - контекст для управления жизненным циклом
        - `cancel context.CancelFunc` - функция отмены контекста
    - Метод `NewWorker()` - создание нового воркера
    - Метод `Start()` - запуск воркера в горутине
    - Метод `Stop()` - остановка воркера
    - Метод `run()` - основной цикл воркера
    - Метод `checkAllSites()` - проверка всех сайтов и сохранение результатов

2. **Расширенный Repository** (`infrastructure/repository/db.go`):
    - `SaveCheck()` - сохранение результата проверки в БД
    - `GetLatestCheck()` - получение последней проверки для сайта
    - `GetCheckHistory()` - получение истории проверок для сайта

### Изменения в существующих компонентах

- **cmd/main.go**:
    - Использование `context.Context` для graceful shutdown
    - Использование `errgroup` для управления горутинами
    - Инициализация и запуск воркера
    - Остановка воркера при завершении работы сервера

## Как работает воркер

1. **Инициализация:**
    - Воркер создается с интервалом проверки (по умолчанию 30 секунд)
    - Создается контекст для управления жизненным циклом

2. **Запуск:**
    - Воркер запускается в отдельной горутине
    - Выполняется немедленная проверка всех сайтов
    - Затем проверки выполняются периодически с заданным интервалом

3. **Цикл проверки:**
    - Получение списка всех сайтов из БД
    - Для каждого сайта:
        - Проверка доступности через `CheckSiteAvailability()`
        - Сохранение результата (статус и время ответа) в БД
        - Логирование результата

4. **Остановка:**
    - При получении сигнала завершения (SIGINT, SIGTERM)
    - Контекст отменяется
    - Воркер корректно завершает работу

## Использование errgroup

Проект использует `golang.org/x/sync/errgroup` для управления горутинами:
- HTTP сервер запускается в одной горутине
- Ожидание сигналов завершения в другой горутине
- Обе горутины управляются через errgroup
- При ошибке в любой горутине все остальные корректно завершаются

## Интервал проверки

По умолчанию воркер проверяет сайты каждые **30 секунд**. Это значение можно изменить в `cmd/main.go`:

```go
checkInterval := 30 * time.Second
```

## Логирование

Воркер логирует:
- Запуск воркера с указанием интервала
- Начало проверки всех сайтов
- Результат проверки каждого сайта (статус и время ответа)
- Завершение проверки всех сайтов
- Остановку воркера

## Graceful Shutdown

При остановке сервера:
1. Получение сигнала завершения (SIGINT или SIGTERM)
2. Отмена контекста
3. Остановка воркера
4. Закрытие соединения с БД
5. Завершение работы сервера

## Зависимости

- `golang.org/x/sync/errgroup` - для управления горутинами

## Как запустить

1. Установите зависимости:
```bash
go mod tidy
```

2. Запустите сервер:
```bash
go run ./cmd/main.go
```

3. Воркер автоматически начнет проверять сайты:
    - Сразу после запуска выполнится первая проверка
    - Затем проверки будут выполняться каждые 30 секунд
    - Результаты сохраняются в таблицу `checks` в БД

## Проверка работы воркера

1. **Добавьте сайт через API:**
```bash
curl -X POST http://localhost:8080/api/site \
  -H "Content-Type: application/json" \
  -d '{"url": "https://example.com"}'
```

2. **Проверьте логи:**
    - В консоли должны появиться сообщения о проверке сайтов
    - Каждые 30 секунд будет выполняться новая проверка
